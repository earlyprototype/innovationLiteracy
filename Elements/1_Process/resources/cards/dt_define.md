# Define

## What Happens Here

You close the first diamond. All the raw material from Discovery gets synthesised into a clear problem statement. This is the hardest intellectual work in the entire process — and the most commonly rushed.

**The discipline:** A good problem statement opens possibilities. A bad one closes them before you've started.

## The Craft of Problem Framing

**"How Might We..." (HMW) statements** are the standard tool. They work because they balance constraint with openness:

- Too broad: *"HMW improve the employee experience?"* — no direction
- Too narrow: *"HMW build a better onboarding app?"* — solution already embedded
- Right: *"HMW help new employees build trusted relationships in their first 30 days?"* — clear constraint, open solution space

## The Solution-Embedded Problem Statement

The #1 failure mode in Define. Watch for problem statements that are actually solutions in disguise:

| Sounds like a problem | Actually a solution |
|----------------------|---------------------|
| "We need a dashboard" | Dashboard is the solution — what's the problem it solves? |
| "Users need faster onboarding" | Faster assumes the fix — maybe the issue is confidence, not speed |
| "We lack a feedback mechanism" | Mechanism is a solution category — what unmet need does it address? |

**Test:** If your problem statement could only be solved one way, it's a solution wearing a problem's clothes.

## Point of View (POV) Statement

A sharper tool than HMW for complex problems:

**[User]** needs to **[need]** because **[insight]**

The "because" forces you to articulate the *why* — the insight from Discovery that makes this framing uniquely informed rather than generic.

## Common Mistakes

- Framing the problem around organisational convenience rather than user need
- Skipping synthesis to jump straight from interviews to "so we should build..."
- Treating Define as a 30-minute exercise rather than the critical convergence point it is
