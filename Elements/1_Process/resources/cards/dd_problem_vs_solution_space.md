# Problem Space vs Solution Space

## The Hinge

The Double Diamond has four phases but only one critical transition: the moment you move from Problem Space (first diamond) to Solution Space (second diamond). Everything depends on getting this transition right.

**Problem Space** (Discover → Define): *What problem are we solving and for whom?*
**Solution Space** (Develop → Deliver): *How do we solve it well?*

```
  ◇ Problem Space ◇     ◇ Solution Space ◇
  Discover → Define  →  Develop → Deliver
   (open)   (close)      (open)   (close)
                    ↑
              THE HINGE
```

## The #1 Mistake

**Jumping to Solution Space before closing Problem Space.** This is the single most common innovation failure. It looks like:

- "We need an app for..." (solution) before "Users struggle with..." (problem)
- Debating features before agreeing on who the user is
- Prototyping before defining what success looks like
- "The CEO wants a dashboard" — solution dictated, problem never asked

**Why it happens:** Solution space feels productive. Problem space feels uncertain. Organisations reward action, not understanding. The gravitational pull toward "just build something" is constant.

## The Readiness Test

You are ready to cross the hinge when you can answer:

1. **Who** specifically are we solving for? (Named user segment, not "everyone")
2. **What** is their core unmet need? (Observed, not assumed)
3. **Why** does this matter to them? (Their words, not yours)
4. **How** will we know if we've solved it? (Measurable outcome, not opinion)

If any answer is vague, you're not ready. Going anyway doesn't save time — it costs it downstream as rework.

## What Good Looks Like

A well-crossed hinge produces a problem statement that the whole team can recite and that constrains solution generation without predetermining it. The team feels slightly uncomfortable — the problem is clear enough to be challenging but not so narrow that the answer is obvious.

## Common Mistakes

- Treating the hinge as a formality rather than a gate
- Allowing senior stakeholders to skip Problem Space ("I already know the problem")
- Crossing back and forth repeatedly without commitment (analysis paralysis)
- Confusing "we've spent enough time on this" with "we understand the problem"
