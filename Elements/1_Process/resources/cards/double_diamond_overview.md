# The Double Diamond Framework

## Why It Matters

The Double Diamond makes visible what most teams do wrong: they converge too soon, on the wrong problem, with insufficient exploration. Two diamonds, four phases, one critical insight: you must open up (diverge) before you close down (converge), and you must do this twice — once for the problem, once for the solution.

## Two Diamonds, Two Jobs

**First Diamond (Discover → Define):** Ensures you're solving the right problem.
**Second Diamond (Develop → Deliver):** Ensures you're solving it well.

The space between them is the hinge — get the problem wrong and everything downstream is waste, no matter how brilliantly executed.

## Core Principle

**Diverge Before You Converge, Every Time.** You cannot evaluate options you haven't generated. You cannot select the best path if you've only considered one. Opening the diamond generates raw material; closing it selects from that material.

## The Visual Does the Work

When a team argues about features before agreeing on the problem, you don't need to lecture — point at the diagram and ask: "Which diamond are we in?"

```
    Discover  →  Define  →  Develop  →  Deliver
   (diverge)  (converge)  (diverge)  (converge)

Problem Space ◇         Solution Space ◇
```

## Practical Application

- **Print it large, keep it visible:** When teams get lost, they can locate themselves visually
- **Interrupt jumps between diamonds:** "That's a great solution — let's hold it. We're still in problem space." Write it down, park it visibly, return to exploration
- **Ban evaluation during divergence:** "No buts" on a visible sign if required
