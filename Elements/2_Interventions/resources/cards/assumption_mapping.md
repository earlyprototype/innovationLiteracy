# Assumption Mapping: Making the Implicit Testable

## Why It Matters

Every solution contains hidden assumptions — beliefs treated as facts because they've never been questioned. Assumption mapping forces teams to extract implicit assumptions, make them explicit, assess their risk, and test them before committing resources.

It's a deceptively simple intervention with profound impact: writing down what must be true transforms wishful thinking into testable hypotheses.

## The Four Categories

| Category | Question | Examples |
|----------|----------|----------|
| **Desirability** | Do people actually want this? | "Users find the current process painful enough to adopt a new one" |
| **Feasibility** | Can we actually build this? | "The API integration is possible within our constraints" |
| **Viability** | Does this work as a business? | "Users will pay enough to cover costs" |
| **Adaptability** | Will the org adopt this? | "Staff will adopt new tools/workflows" |

## The Riskiest Assumption Framework

**Risk = Uncertainty × Impact**

| | Low Impact | High Impact |
|---|-----------|-------------|
| **High Uncertainty** | Test if cheap | **Test immediately** |
| **Low Uncertainty** | Safe to proceed | Gather evidence |

## The Process

1. **Generate:** "For this to succeed, what must be true?" Silent individual work, 5 minutes
2. **Cluster:** Group similar assumptions (20-30 → 5-8 core beliefs)
3. **Assess Risk:** Rate uncertainty × impact, plot on 2x2
4. **Prioritise Testing:** Top-right quadrant = test immediately
5. **Design Tests:** "Cheapest, fastest experiment to validate?"

## Key Principles

- **Frame as hypotheses:** "We believe X. To test: Y. Success looks like: Z."
- **Test cheapest first, not everything**
- **Celebrate invalidation** — learning for £5k vs discovering post-launch for £50k
- **Keep the map visible** — mark validated (green), invalidated (red), uncertain (yellow)
